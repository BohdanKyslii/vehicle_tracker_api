# Warehouse Expense Tracker — Загальний опис проєкту

> Документ для review сеньйором. Мета — підтвердити архітектуру,
> уточнити бізнес-логіку та погодити технічний стек перед стартом розробки.
>
> **На відміну від `task_description/transport`** (який є greenfield-дизайном
> нової системи, що не відповідає поточному Django-застосунку `transport`),
> цей документ **описує реальну логіку, що вже реалізована** в Django-застосунку
> `warehouse` (`warehouse/models.py`, `views.py`, `utils.py`), і переносить її
> у окремий JS/TypeScript/Vite фронтенд — зі збереженням бізнес-правил,
> але з виправленням знайдених непослідовностей і додаванням аналітики.

## 1. Бізнес-контекст

Компанія веде облік складських витрат (оренда, комунальні, зарплати
складського персоналу, ремонт обладнання, витратні матеріали тощо)
по кожному складу окремо, з розбивкою на категорії та підкатегорії.

Витрати діляться на два типи:

- **Фіксовані** (оренда, зарплата — сума стала і рідко змінюється,
  але коли змінюється — діє з певної дати і наперед).
- **Змінні** (комунальні, ремонт, витратні матеріали — вводяться вручну
  щомісяця, сума може відрізнятись кожного разу).

Наразі облік ведеться через Django-застосунок з HTML-сторінками
(`warehouse/templates/warehouse/*.html`), без окремого API. Мета —
винести UI в окремий SPA на React/TypeScript/Vite, а Django залишити
як backend + JSON/DRF API, за тим самим підходом, що й для `transport`.

**Цілі системи:**
- Зручне щомісячне внесення витрат (одиночне і масове по всьому складу).
- Коректне, передбачуване ведення історії фіксованих витрат з каскадним
  оновленням вже згенерованих майбутніх записів (наразі це не працює —
  див. розділ 7).
- Аналітика: динаміка витрат, структура по категоріях, порівняння
  складів, план/факт, прогноз — з графіками (Recharts), а не лише
  таблицями (наразі сторінка «Аналіз» — суцільні таблиці без жодного графіка).
- Довідники (склади/категорії/підкатегорії/типи витрат/об'єкти витрат)
  редагуються прямо в SPA, без переходу в Django admin.
- Розміщення на одному сайті з фронтендом `transport` (див. §9 і
  `08_PROJECT_STRUCTURE.md`), розробку якого веде інший розробник.

## 2. Ролі користувачів

Поточна модель прав — по групах Django (`common/views.py: home_selector`):
`can_access_warehouse = request.user.is_superuser or groups includes "Warehouse"`.
Це модуль-рівень доступу (весь `warehouse` або нічого), без ролей усередині
модуля. Пропонується зберегти цю модель на MVP, але явно розрізняти
дії в UI за зручністю:

| Роль (умовна, не окрема group) | Основні дії |
|------|-------------|
| **Комірник / кладовщик** | Щомісячне внесення змінних витрат (форма додавання, масове внесення) |
| **Бухгалтер / фінансист** | Ведення історії фіксованих витрат, перевірка року (year review), імпорт/експорт Excel |
| **Керівник** | Дашборд, аналітика, порівняння складів, план/факт |
| **Адміністратор довідників** | Склади, категорії, підкатегорії, типи витрат, об'єкти витрат |

> Всі ролі належать одній Django-групі `Warehouse` (або superuser) —
> розмежування "хто що робить" відбувається лише на рівні UI-навігації,
> не на рівні прав доступу до API. Питання, чи потрібні реальні
> per-role permissions, винесено в §10.

## 3. Структура даних (ієрархія)

```
Warehouse (склад)
  └── ExpenseObject (об'єкт витрат) ── унікальний в межах
        ├── Category (категорія)         (warehouse, category,
        ├── Subcategory (підкатегорія)     subcategory, name)
        ├── CostType (тип витрат: Фіксована / Змінна)
        └── name (довільна назва)

ExpenseObject ──< ExpenseRecord (запис витрат: рік + місяць + сума)
                     UNIQUE (expense_object, year, month)

ExpenseObject ──< FixedExpenseHistory (тільки для «Фіксована»)
                     ефективна дата (start_date) + сума
                     — паралельний журнал "яка сума мала б діяти"
```

`Category` і `Subcategory` — спільні довідники (не прив'язані до
конкретного складу), `ExpenseObject` — це вже конкретна комбінація
склад + категорія + підкатегорія + назва + тип витрат, до якої
прив'язуються місячні суми.

## 4. Фіксовані vs Змінні витрати — точна бізнес-логіка

### 4.1 Як визначається "фіксована" (поточна реалізація)

`CostType` — вільна довідникова таблиця (просто `name`), без окремого
enum/boolean. Уся система визначає "фіксована" через порівняння рядка:
`expense_object.cost_type.name == "Фіксована"`. Це крихко (typo в назві
зламає логіку) — **пропонується покращення**: додати `CostType.code`
(`'fixed' | 'variable'`), зберігши `name` для відображення (див.
`02_DATABASE_SCHEMA.md`).

### 4.2 Автогенерація фіксованих записів при додаванні змінної витрати

Поточна логіка (`add_expense_record` → `copy_fixed_expenses_to_month`):

```
Коли користувач зберігає НОВИЙ ЗМІННИЙ запис на місяць M:
  Для кожного ExpenseObject з cost_type = "Фіксована":
    Якщо ExpenseRecord(object, рік, M) вже існує → пропустити
    Інакше:
      Знайти останній FixedExpenseHistory зі start_date <= 1-е число M
      Якщо знайдено → створити ExpenseRecord(
        amount = history.amount,
        is_auto_generated = True,
        comment = "Автоматично створено. Актуальна сума з {start_date}"
      )
      Якщо не знайдено → нічого не створювати (об'єкт лишається без запису)
```

Тобто фіксовані записи місяця "з'являються" лише як побічний ефект
додавання будь-якої змінної витрати за цей місяць — неочевидна поведінка
для користувача. **Покращення**: явна кнопка «Згенерувати фіксовані
записи за місяць» на сторінці масового внесення, і той самий виклик
залишити як фоновий safety-net.

### 4.3 Каскадне оновлення при зміні фіксованої суми — НЕ ПРАЦЮЄ (баг)

У `utils.py` є повністю готова функція `update_fixed_expense()`, яка:
1. Оновлює/створює `ExpenseRecord` на місяць, з якого діє нова сума.
2. Каскадно перезаписує суму в усіх **вже згенерованих автоматично**
   (`is_auto_generated=True`) записах на майбутні місяці цього ж об'єкта
   (записи, введені вручну — не чіпає, "факт має пріоритет").

Але `fixed_expenses_management` (POST) **її не викликає** — сторінка лише
дописує новий рядок в `FixedExpenseHistory`, і все. Це створює розбіжність
між "довідником історії" та фактичними місячними записами: старі
автозгенеровані записи на майбутні місяці лишаються зі старою сумою,
доки хтось вручну не перегенерує.

**Рішення для нової системи**: при збереженні нового запису
`FixedExpenseHistory` — одразу викликати каскадне оновлення (аналог
`update_fixed_expense`), показавши користувачу підтвердження
(«Буде оновлено N вже згенерованих записів на майбутні місяці»)
перед застосуванням.

### 4.4 Визначення "якої суми фіксованого об'єкта" для конкретного місяця

Використовується у формі масового внесення (`get_expense_objects_by_warehouse`,
наразі не підключена до жодного URL — див. §7.1). Пріоритет:

```
1. Остання історія зі start_date <= 1-е число цільового місяця
     → статус "current" ("Актуальна з {дата}")
2. Якщо немає — найближча історія зі start_date > 1-е число місяця
     → статус "future" ("Майбутня з {дата}") — показується, але
       НЕ підставляється як значення за замовчуванням
3. Якщо немає — остання історія взагалі (найновіша за start_date)
     → статус "fallback" ("Остання: {дата}")
4. Якщо історії взагалі немає
     → статус "none" ("Немає записів в історії") — поле стає
       редагованим вручну (єдиний виняток, коли фіксований об'єкт
       можна ввести напряму)
```

### 4.5 Правила нуля / видалення

Поточна поведінка непослідовна між двома формами:

| Форма | Поведінка при 0 / порожньо |
|-------|----------------------------|
| Масове внесення за місяць (`save_monthly_expenses`) | Фіксовані: **пропускається** (не зберігається). Змінні: 0 дозволено зберегти. |
| Перевірка року, грід 12 міс (`update_monthly_expenses`) | **Видаляє** існуючий `ExpenseRecord`, якщо значення ≤ 0 / порожнє — для будь-якого типу. |

**Покращення**: явна кнопка "Очистити запис" замість неявного видалення
через 0, і заборона (з поясненням у UI) зберігати 0 для фіксованого
типу — фіксовану суму можна змінити лише через `FixedExpenseHistory`.

## 5. Довідники

### 5.1 `Warehouse` (склад)
Просто `name` (унікальне). Немає `is_active` — **пропонується додати**
для архівації без видалення (складів, що закрились), за аналогією з
іншими застосунками проєкту.

### 5.2 `Category` / `Subcategory`
`Category.name`, `Subcategory.name` + FK на `Category`. Спільні для
всіх складів.

### 5.3 `CostType`
`name` (`"Фіксована"` / `"Змінна"` — фактично будь-які рядки, ніхто це
не обмежує). **Пропонується додати `code`** (`fixed`/`variable`) з
`unique=True`, щоб прибрати крихке порівняння рядків.

### 5.4 `ExpenseObject` (об'єкт витрат)
Комбінація `warehouse + category + subcategory + name + cost_type`,
`unique_together = (warehouse, category, subcategory, name)`. Наразі
редагується **тільки через Django admin** (кнопка «Додати об'єкт» на
`/expense-objects/` веде прямо в `admin:warehouse_expenseobject_add`).
**Покращення**: повноцінна CRUD-сторінка в SPA.

## 6. Записи витрат

### `ExpenseRecord`
`expense_object`, `year`, `month` (1–12), `amount` (Decimal ≥ 0),
`comment`, `is_auto_generated` (bool), `base_record` (self-FK,
**задекларовано, але ніде не використовується — мертве поле**,
прибрати або задіяти для реального трасування "згенеровано з"),
`created_at`, `updated_at`. `unique_together = (expense_object, year, month)`.

### `FixedExpenseHistory`
`expense_object`, `amount`, `start_date` (дата, з якої діє сума),
`comment`, `created_at`, `created_by` (звичайний `CharField`, не FK на
User — **пропонується зробити FK на `auth.User`** зі збереженням
текстового снепшоту імені для історичності).

## 7. Відомі проблеми поточної реалізації (врахувати в новій системі)

1. **Непідключений ендпоінт**: `get_expense_objects_by_warehouse(request, warehouse_id)`
   очікує `warehouse_id` як частину шляху, але в `urls.py` зареєстровано
   лише `api/expense-objects/<int:id>/` → веде на **інший** view
   (`get_expense_object_api`, одиночний об'єкт за `ExpenseObject.id`).
   Кнопка «Завантажити витрати» на `/monthly-expenses/` фактично не
   працює в поточному застосунку. У новому API це має бути окремий,
   явно названий endpoint (`GET /api/warehouses/{id}/expense-objects/?year=&month=`).
2. **`update_fixed_expense` не викликається** — див. §4.3.
3. **Рядкове порівняння** `cost_type.name == "Фіксована"` замість enum/коду — див. §4.1.
4. **`base_record`** — мертве поле в `ExpenseRecord`.
5. **Редіректи без namespace** (`add_expense_record`, `delete_expense_record`
   роблять `redirect("expenses_list")` замість `redirect("warehouse:expenses_list")`)
   — суто Django-деталь, але сигналізує про відсутність тестового покриття
   (`warehouse/tests.py` — порожній, 0 тестів).
6. **Валідація GET-параметрів** — лише `dashboard` перевіряє, що `year`
   є числом; інші view впадуть у 500 на некоректному вводі. Новий API
   має уніфіковано валідовувати (400, а не 500).
7. **`analysis.html`** рахує реальний `percentage`, але в шаблоні
   прогрес-бар має захардкоджену ширину `10%` — візуально невірно.
8. **Немає жодного графіка на сторінці аналізу**, хоча дашборд вже
   використовує Chart.js (лінійний + doughnut) — аналітика лишається
   суто табличною.
9. **Дублювання довідника місяців**: `ExpenseRecordStepForm.month` хардкодить
   україномовний список місяців, хоча `common.constants.MONTH_CHOICES`
   вже існує і використовується в інших місцях.
10. **Немає DRF / жодного API-фреймворку** в проєкті — усі поточні JSON
    endpoint'и — це `JsonResponse` вручну, без пагінації, без єдиного
    формату помилок. Для нового SPA потрібен новий шар API (детальніше
    в §9 і `06_IMPLEMENTATION_PLAN.md`).

## 10. Питання для сеньйора

1. **Каскадне оновлення фіксованих витрат** (§4.3) — застосовувати
   автоматично одразу при збереженні нової `FixedExpenseHistory`,
   чи через окрему кнопку підтвердження з попереднім переглядом?
2. **`CostType.code`** — додавати нове поле міграцією зараз, чи
   тимчасово мапити `name` → `code` на рівні API-серіалізації без
   зміни БД?
3. **Права доступу** — лишити єдину групу `Warehouse` на весь модуль,
   чи вводити реальні ролі (комірник / бухгалтер / керівник) з різними
   правами на API?
4. **CRUD об'єктів витрат у SPA** — повністю замінює Django admin,
   чи лишити admin як fallback для рідкісних масових правок?
