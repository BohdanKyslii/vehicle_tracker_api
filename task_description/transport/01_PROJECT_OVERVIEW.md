# Vehicle Cost Tracker — Загальний опис проєкту

> Документ для review сеньйором. Мета — підтвердити архітектуру,
> уточнити бізнес-логіку та погодити технічний стек перед стартом розробки.

---

## 1. Бізнес-контекст

Компанія має автопарк вантажних авто (доставка напоїв/товарів).
Наразі облік витрат ведеться вручну в Excel — немає оперативних даних
про пробіги, витрати палива, транспортну собівартість по накладних.

**Цілі системи:**
- Автоматизувати щоденний збір даних від водіїв (пробіг, паливо, накладні).
- Розраховувати повну собівартість доставки по кожній накладній і товару.
- Охопити всі канали доставки: власний автопарк, найманий транспорт, служби доставки.
- Надати менеджменту аналітичні дашборди по авто, маршрутах, витратах.
- Підготувати дані для майбутньої інтеграції з 1С/BAS.

## 2. Ролі користувачів

| Роль | Пристрій | Основні дії |
|------|----------|-------------|
| **Водій** | Смартфон (PWA) | Щоденний пробіг, паливо, сканування QR накладних, повернення, додатковий вантаж |
| **Логіст** | ПК | Налаштування авто та режимів; місячні витрати; внесення рейсів найманого транспорту |
| **Менеджер-операціоніст** | ПК | Щотижневий імпорт реєстру накладних із 1С; матчинг повернень; реєстри служб доставки |
| **Керівник** | ПК / планшет | Аналітичні дашборди, фільтри, звіти |

## 3. Канали доставки

Кожна накладна належить **рівно одному** каналу. Канал визначається при скануванні
або внесенні вручну. Подвійне призначення виключено на рівні бізнес-логіки.

| Канал | Хто вносить | Модель витрат |
|-------|------------|---------------|
| **Власний автопарк** | Водій (PWA) | Місячні витрати на авто, розподілені по накладних |
| **Найманий транспорт** | Логіст (ПК) | Фактична вартість рейсу, прив'язана до накладних |
| **Служби доставки** | Менеджер (ПК) | Реєстри витрат від НП / Міст Експрес |

## 4. Два режими трекінгу водія (власний автопарк)

Логіст задає **дефолтний режим** на рівні авто.
Водій може **змінити режим на поточний день** самостійно перед початком маршруту.

### Режим `daily` — щоденний мінімум

Підходить для авто, де не потрібна деталізація по точках вивантаження.

```
Щоранку (один запис на день):
  ├── Одометр поточного ранку *
  ├── Заправка (л + грн) — опційно
  ├── AdBlue (л + грн) — опційно
  ├── Інші витрати (грн + коментар) — опційно
  ├── Повернення — опційно (див. п. 4.3)
  ├── Додатковий вантаж — опційно (див. п. 4.4)
  ├── Кількість палет *
  └── Сканування QR накладних поточного дня *

Логіка QR-сканування (daily):
  └── Перша відскана накладна → запитуємо «Це накладна точки вивантаження?»
      → підтверджуємо — далі сканування без запитів
  └── На одному вантажі може бути 3 палети і 6 накладних на різні адреси

Пробіг за вчора = одометр сьогодні − одометр вчора
```

### Режим `full` — повний маршрут

Підходить для авто, де важлива деталізація по точках вивантаження.

```
Ранок (склад):
  ├── Одометр *
  └── Сканування QR накладних на поточний день *

Після кожного вивантаження:
  ├── Одометр *
  ├── Кількість палет на точці *
  ├── Сканування QR накладної точки *
  ├── Сканування інших накладних цієї точки (якщо є)
  ├── Відмова від поставки — опційно:
  │   ├── Повна відмова (відмітка на всю накладну)
  │   └── Часткова: артикул + кількість + коментар
  ├── Повернення — опційно (див. п. 4.3)
  └── Додатковий вантаж — опційно (див. п. 4.4)

У будь-який момент:
  ├── Заправка (л + грн)
  ├── AdBlue (л + грн)
  └── Інші витрати (грн + коментар)

Вечір (парковка):
  └── Одометр кінця маршруту *

Дає додатково:
  ├── Пробіг між точками вивантаження
  ├── Порожній пробіг (остання точка → склад наступного ранку)
  └── Час між вивантаженнями
```

### 4.3 Подія «Повернення»

Доступна в обох режимах. Водій вносить номер накладної клієнта вручну.
Матчинг виконує **менеджер-операціоніст** (номер оригінальної накладної
клієнта зазначений у полі «коментар» накладної на повернення).

| Поле | Обов'язковість |
|------|---------------|
| Номер накладної клієнта | Обов'язково |
| Дата | Автоматично (поточна) |

### 4.4 Подія «Додатковий вантаж»

Доступна в обох режимах. Фіксує нетипові перевезення (не з основного реєстру).

| Поле | Обов'язковість |
|------|---------------|
| Номер накладної клієнта | Опційно (може бути `null`) |
| Звідки везуть товар | Обов'язково |
| Куди везуть товар | Обов'язково |
| Вага вантажу (кг) | Обов'язково |
| Коментар | Опційно |
| Дата | Автоматично (поточна) |

## 5. Витрати — структура і джерела

### 5.1 Щоденні (від водія, власний автопарк)

| Стаття | Джерело |
|--------|---------|
| Паливо (л) | Водій по чеку |
| Паливо (грн) | Водій по чеку |
| AdBlue (л) | Водій по чеку |
| AdBlue (грн) | Водій по чеку |
| Інші витрати (грн + коментар) | Водій, довільно |

### 5.2 Місячні витрати авто (від логіста)

Логіст вносить дані один раз на місяць по кожному авто власного автопарку.

| Поле форми | Тип | Примітка |
|------------|-----|---------|
| Авто | Select із `cars` | |
| Місяць | Month picker | |
| ЗП водія (грн) | Number | |
| Податки із ЗП (грн) | Number | |
| Амортизація авто (грн) | Number | Стала величина, може коригуватись |
| Ремонт / ТО фактичний (грн) | Number | Якщо порожньо — рахується `2 грн × км` |
| Ставка ремонту (грн/км) | Number | Default: 2.00 |
| Інші витрати (грн) | Number | |
| Коментар | Text | |

> **Логіка ремонту:** якщо логіст не вніс фактичні витрати — система рахує
> `ставка × км за місяць`. Фактичні мають пріоритет.

### 5.3 Найманий транспорт (від логіста)

Логіст вносить кожен рейс найманого авто окремо.

| Поле форми | Тип | Примітка |
|------------|-----|---------|
| Номер авто | Text | Вільний ввід (не з довідника) |
| Назва маршруту | Text | Довільна, напр. «Пирятин, Полтава, Харків» |
| Дата рейсу | Date | |
| Кількість палет | Number | |
| Вартість доставки (грн) | Number | Фактична, від перевізника |
| Коментар | Text | Опційно |
| Накладні | QR-скан або ручний ввід | Прив'язуються до рейсу |

> Накладні найманого транспорту **не можуть** перетинатись із накладними
> власного автопарку або служб доставки.

### 5.4 Служби доставки (Нова Пошта, Міст Експрес тощо)

Менеджер вносить накладні та імпортує реєстр витрат від служби.

**Сторінка прив'язки накладних до служби:**

| Поле | Тип | Примітка |
|------|-----|---------|
| Служба доставки | Select | НП / Міст Експрес / інша |
| Дата відправки | Date | |
| ТТН (номер відправлення) | Text | Номер у службі доставки |
| Накладна | QR-скан або ручний ввід | Одна накладна = одне відправлення (або декілька) |

**Імпорт реєстру витрат від служби:**

| Поле | Тип | Примітка |
|------|-----|---------|
| ТТН | Text | Матч по ТТН |
| Вага відправлення (кг) | Number | |
| Вартість доставки (грн) | Number | |
| Дата | Date | |

> Накладні служб доставки **не можуть** перетинатись із накладними
> власного автопарку або найманого транспорту.

### 5.5 Реєстр накладних (від менеджера, щотижня)

Файл CSV / Excel із 1С. Містить відвантаження та повернення (від'ємна кількість).
Три юридичні особи: **ESP**, **OPT**, **Rubin**.

| Поле в 1С | Тип | Примітка |
|-----------|-----|---------|
| `legal_entity` | `VARCHAR(10)` | ESP / OPT / Rubin |
| `waybill_date` | `DATE` | Дата накладної |
| `waybill_number` | `VARCHAR(50)` | Номер накладної |
| `customer_id` | `VARCHAR(50)` | ID клієнта |
| `store_id` | `VARCHAR(50)` | ID торгової точки (магазину) |
| `product_id` | `VARCHAR(50)` | Артикул товару |
| `quantity` | `NUMERIC` | **+** відвантаження; **−** повернення |
| `price_uah` | `NUMERIC` | Ціна за одиницю |
| `total_uah` | `NUMERIC` | Сума рядка |
| `line_position` | `SMALLINT` | Номер позиції в накладній |
| `comment` | `TEXT` | Для повернень — номер накладної клієнта |

**Унікальний ключ рядка:** `waybill_number + line_position`


## 8. Аналітика

### По авто (власний автопарк)
- Пробіг за день / місяць (завантажений / порожній — тільки `full`)
- Витрати палива (л / 100 км), загальні витрати (грн / км)
- Кількість палет, вага та об'єм доставленого вантажу

### По накладній
- Транспортна собівартість (залежно від каналу доставки)
- % транспортних витрат від суми продажу
- Вага та об'єм вантажу

### По клієнту / магазину
- Транспортна собівартість, % від продажу
- Розбивка по каналах доставки
- Загальна вага та об'єм

### По товару / категорії
- Транспортна собівартість одиниці
- Вага / об'єм / об'ємна вага

### По каналу доставки
- Власний автопарк vs найманий транспорт vs служби доставки
- Порівняльна вартість грн/кг або грн/палета

### По маршруту (тільки `full`, власний автопарк)
- Пробіг між точками, час між точками, порожній пробіг

## 10. Питання для сеньйора

1. **Розподіл місячних витрат по накладних** — пропорційно до суми продажу,
   ваги, кількості палет? Або комбінований коефіцієнт?
2. **Прив'язка накладної до авто** — через QR від водія чи через реєстр 1С?
   Що є пріоритетом при конфлікті?
3. **Одометр** — довіряємо водію чи потрібна валідація (GPS, фото)?
4. **Імпорт із 1С** — CSV вручну або планується API / автоматична синхронізація?
5. **Права доступу** — чи потрібна авторизація в MVP, чи достатньо url-based?
6. **Зміна режиму водієм** — чи фіксувати в журналі?
7. **Матчинг повернень** — окремий UI або тільки Excel?
8. **Адміністрування довідників** — окремий розділ або пряме редагування БД?
9. **Юридична особа в аналітиці** — окремо по ESP / OPT / Rubin чи зведено?
10. **Ексклюзивність каналів** — як технічно блокувати дублювання накладних
    між каналами? Constraint на рівні БД або тільки валідація на UI?
11. **Служби доставки** — чи є єдиний формат реєстру витрат від НП і Міст Експрес,
    чи потрібен окремий парсер для кожного?
12. **Палети** — кількість палет на рейс чи на точку вивантаження (у `full` режимі)?
